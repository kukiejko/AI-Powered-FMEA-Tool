📖 User Guide

This tool helps you identify, assess, and track failure modes in your processes or equipment. Choose between two modes when creating a project, based on your team's needs and time available.

⚡ Simplified Mode

Best for quick risk assessments, small teams, or first-time FMEA users. Covers the essentials: identify what can go wrong, rate the risk, and assign corrective actions.

  • Core columns: Process Step, Failure Mode, Effect, Cause, S / O / D, RPN
  • Action tracking: owner, due date, % complete
  • Incident linking and count
  • AI-powered failure mode generation from uploaded documents
🔬 Advanced Mode

Full AIAG/VDA-aligned workflow for formal engineering teams. Adds deeper analysis columns and post-action re-assessment to close the full risk loop.

  • Everything in Simplified, plus:
  • Action Priority (AP): H / M / L per AIAG/VDA
  • S×O: severity-occurrence exposure index
  • Prevention & Detection actions split
  • Action Status (Open → Completed)
  • Revised S / O / D and RPN after action
  • Source traceability and Notes
1
Create a project — click + New Project on the dashboard, enter a name, and choose Simplified or Advanced mode.
2
Add failure modes — upload a document and click Analyze & Generate FMEA to let AI extract failure modes, or add rows manually.
3
Rate each failure — score Severity (S), Occurrence (O), and Detection (D) from 1–10. RPN = S × O × D is calculated automatically.
4
Assign actions — for high-RPN items, describe corrective actions, assign an owner, and set a due date.
5
Log incidents — use the Incidents tab to record real-world failures. Link them to FMEA rows to validate your Occurrence ratings.
6
Track & export — monitor % complete per row. Export to CSV for reporting or import JSON to continue work later.
💡 Tip: Data is saved automatically to your browser. Use Export JSON regularly to back up your work, and Import JSON to restore it.

⚡ Simplified Mode — Step by Step

Simplified mode focuses on the core FMEA loop: identify failures → rate risk → act. Every column is essential; nothing is optional noise.

Column
What to enter
Tips
#
Auto-assigned row number
Used for referencing rows in discussions and reports
Process Step
The step, subsystem, or function being analysed
Be specific — e.g. "Gas flow control valve" not just "Valve"
Failure Mode
How the step could fail — the physical failure
e.g. "Valve stuck closed", "Seal leak", "Sensor drift"
Effect
What happens to the customer / process if this failure occurs
Think downstream impact: "Process stops", "Yield loss >30%"
Cause
Root cause of the failure mode
Use "5 Whys" thinking — go one level deeper than the obvious
S (Severity)
Rate 1–10: how bad is the effect if it happens?
10 = safety/regulatory, 7 = major damage, 4 = minor, 1 = no effect. Never lower S just because a fix exists.
O (Occurrence)
Rate 1–10: how often does this cause occur?
Use MTBF data or incident count if available. Incident badges help calibrate this score.
D (Detection)
Rate 1–10: how hard is it to detect before impact?
10 = impossible to detect, 1 = automatic alarm always fires. Lower is better.
RPN
S × O × D — calculated automatically
<50 Low 50–99 Med 100–199 High ≥200 Critical
Inc.
Count of real incidents linked to this row
Higher count = real-world evidence that O score may need to increase
Action
Corrective or preventive action to reduce risk
Focus on reducing S (redesign), O (prevention), or D (detection improvement)
Action Owner
Person responsible for implementing the action
Must be a named individual, not a department
Action Due Date
Target completion date
Set realistic dates aligned with your project milestones
% Complete
Progress on the action (0–100%)
Update as the action progresses. Bar turns green at 100%.
Incidents
INC-# badges for linked incidents
Click any badge to jump to the Incidents tab. ⚠️ button on each row opens a new incident pre-linked to that row.
💡 When to act: Prioritise rows with RPN ≥ 100 (High) or any row with S ≥ 9 regardless of RPN — a catastrophic but rare failure still warrants a plan.

🔬 Advanced Mode — Step by Step

Advanced mode follows the AIAG & VDA FMEA Handbook 7-step process. It adds Action Priority (AP), which is the AIAG/VDA replacement for RPN as the primary decision tool, plus post-action re-assessment to close the risk loop.

Additional columns (beyond Simplified)

Column
What to enter
Tips
S×O
Severity × Occurrence — auto-calculated
Highlights high-risk exposure independent of detection.
AP
Action Priority: H / M / L — auto-calculated per AIAG/VDA
H = must act. M = should act. L = review and consider.
Source
Document and page that identifies this failure mode
Supports audit traceability.
Prevention Action
Action that prevents the failure cause from occurring
Targets O reduction.
Detection Action
Action that improves ability to detect the failure
Targets D reduction.
Status
Select from: Open / Decision Pending / Implementation Pending / Completed / Not Implemented
An FMEA is not complete until every row is either Completed or Not Implemented with justification.
rS / rO / rD
Revised S, O, D scores after action is implemented
Only fill in once action Status = Completed.
rRPN
Revised RPN = rS × rO × rD — auto-calculated
Compare to original RPN to confirm the action was effective.
Notes
Internal team comments, context, or "pointer to evidence"
Reference test reports, photos, or decision papers.
⚠️ Important: AP gives more weight to Severity than RPN does. A failure with S=9, O=1, D=1 (RPN=9) still gets AP=H because the consequence is catastrophic.

⚠️ Incident Log — How It Works

The Incident Log captures real-world failures and near-misses. Linking incidents to FMEA rows gives you evidence-based data to validate and recalibrate your Occurrence (O) scores over time.

1
Log an incident — go to the ⚠️ Incidents tab and click + Add Incident. Or click the ⚠️ button on any FMEA row to open the form pre-linked to that row.
2
Fill in the form — Timestamp and Description are mandatory. File attachment and FMEA row link are optional but recommended.
3
Read the FMEA table — linked rows show an ⚠️ INC-# badge and a coloured count bubble.
4
Use incidents to calibrate O scores — if a row has 3+ real incidents, your original O score may be too optimistic.
5
Export — use ⬇ CSV to export all incidents with their linked FMEA data for reporting.
💡 Each incident has a unique INC-# number that never changes.
